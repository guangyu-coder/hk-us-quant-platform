# 市场股票池持续扩容设计

日期：2026-04-20

## 背景

当前市场模块已经完成了几轮关键建设：

- `美股 / 港股` 分市场展示
- `全部 / 涨幅榜 / 跌幅榜`
- 市场三分页：
  - 榜单
  - 股票曲线
  - 订单簿
- `market_symbols` 主数据表
- `market_movers_snapshots` 榜单快照
- movers 覆盖率展示
- 准实时榜单刷新

这些能力已经让市场页从“示例行情页”变成了“可用的研究入口”。

但当前还有一个非常明显的瓶颈：

**股票数量仍然不够大，离用户期待的“接近全市场”还有明显差距。**

当前不足主要体现在：

1. `market_symbols` 虽然已经引入数据库主数据，但导入规模仍然偏小
2. 真实行情覆盖率在港股还可以，在美股上仍然偏弱
3. 榜单虽然已经是全量快照模式，但“全量”仍然受制于主数据覆盖范围
4. 当前扩容仍然偏手工，缺少长期可持续的主数据更新链路

所以，下一阶段的重点不应该再是榜单页面表现层，而应该回到数据底盘本身：

**继续扩大股票池，建立更稳的主数据扩容与更新机制。**

## 目标

本次设计目标是：

1. 继续扩大美股与港股股票池覆盖范围
2. 第一阶段仍然只覆盖：
   - `Common Stock`
   - `ETF`
3. 把当前“手工扩容”升级成“可重复执行的主数据导入机制”
4. 保持：
   - 目录离线更新
   - 行情在线获取
   - 榜单快照独立生成
5. 让市场页中的：
   - `全部股票`
   - `涨幅榜`
   - `跌幅榜`
   都建立在更大规模的主数据底座之上

## 非目标

本次不做：

- 权证
- 牛熊证
- REIT
- 优先股
- ADR 单独分类治理
- 全球多市场扩展
- 分钟级或流式全市场行情排名
- 自选股系统
- 复杂企业行为处理（拆股、分红、并购）的研究级精修

当前重点是：

**先把美股和港股的普通股票与 ETF 覆盖面继续做大。**

## 问题判断

当前“股票数太少”的根因，不在前端，而在三层数据链：

### 1. 主数据层覆盖还不够

虽然已经有：

- `market_symbols`
- 导入脚本
- 本地目录文件

但目前主数据仍然更像“扩大的起步版本”，还不是长期可持续的市场目录。

### 2. 行情层成功率限制榜单上限

即使股票目录数量扩大，如果：

- 某一批 symbol 没有真实 quote
- provider 覆盖率不足
- 批量拉取成功率低

那么 movers 快照最终覆盖的股票数仍然上不去。

### 3. 当前扩容链路还偏人工

目前可以扩，但还不够工程化：

- 扩容来源不够统一
- 更新频率不够制度化
- 缺少更清晰的失败统计和导入结果摘要

## 方案比较

### 方案 A：继续扩大本地目录 + 规范导入流程

推荐作为下一阶段。

做法：

- 继续扩充本地 `US / HK` 目录文件
- 统一通过导入脚本写入 `market_symbols`
- 为导入过程增加统计、覆盖摘要和可重复执行入口

优点：

- 实现稳定
- 与现有主数据架构完全兼容
- 风险最低
- 能快速继续提升数量

缺点：

- 目录仍然需要维护
- 真正意义上的“自动全市场同步”还没做到

### 方案 B：直接切到外部源动态获取完整市场目录

优点：

- 理论上最接近真正全市场

缺点：

- 依赖外部源稳定性
- 易受限流和格式变更影响
- 会让市场目录能力变得更脆弱

### 方案 C：目录层混合模式

做法：

- 本地目录保底
- 外部源增量补充
- 导入时去重合并

优点：

- 上限更高
- 未来扩展性更强

缺点：

- 第一阶段复杂度更高
- 去重、优先级、清洗逻辑会更重

## 最终设计

### 1. 扩容策略

下一阶段采用：

**本地目录持续扩大 + 导入脚本标准化 + 榜单真实覆盖率继续提升**

也就是说：

- 目录层先继续做大
- 导入链路先工程化
- 再逐步补行情覆盖率

### 2. 主数据覆盖范围

本阶段继续限制在：

- `US / Common Stock`
- `US / ETF`
- `HK / Common Stock`
- `HK / ETF`

目录扩容后，市场页应当支持：

- 更大的全市场分页浏览
- 更大的榜单候选池
- 更完整的搜索范围

### 3. 主数据来源形态

主数据继续以本地目录文件为核心来源，例如：

- `data/market_symbols.us.json`
- `data/market_symbols.hk.json`

但要求从“静态文件”升级成“受控导入资产”：

- 目录文件需要保持统一结构
- 导入脚本要支持重复执行
- 导入结果要输出统计摘要

### 4. 导入脚本能力增强

现有导入脚本需要继续增强，目标是让它从“能导入”变成“适合长期扩容”。

建议补强：

- 输出导入总数
- 输出新增数
- 输出更新数
- 输出停用数
- 输出 market / instrument_type 分组统计
- 输出异常 symbol 列表摘要
- 支持 dry-run
- 支持只导入某个市场或某个品类

这样我们每次扩容后都能清楚知道：

- 这次到底导了多少
- 哪些被更新了
- 哪些被忽略了

### 5. 市场列表接口策略

市场列表接口下一阶段原则不变：

- `全部股票` 走 `market_symbols`
- 分页浏览
- 搜索仍基于主数据表

但在体验上需要强化：

- 返回更明确的 `total`
- 前端展示市场总股票数
- 当前页显示更清楚

### 6. 榜单快照策略

榜单仍然继续沿用：

- 批量真实行情抓取
- 写入 `market_movers_snapshots`
- 页面只读快照

但下一阶段重点是：

- 让快照候选池来自更大的 `market_symbols`
- 让覆盖率统计更真实反映扩容结果

也就是说，扩容完成后，页面应该能看到：

- 市场总股票数变大
- movers 候选范围变大
- `coverage.covered / coverage.total` 更接近真实目标

### 7. 真实行情覆盖率提升方向

股票数量扩上去之后，下一层问题就会更加明显：

**真实 quote 能否覆盖这些股票。**

所以这一轮设计明确把行情覆盖率列为扩容后的第二优先级：

- 港股继续利用当前覆盖较好的真实源
- 美股需要继续评估更稳定的数据源或批量抓取路径
- movers 快照仍然坚持：
  - 不写模拟数据
  - 只写真实成功返回的 quote

### 8. 页面反馈优化

市场页需要继续强化“数量”和“覆盖率”的感知：

- 当前市场总股票数
- 当前分页区间
- 当前 movers 覆盖数量
- 当前 movers 成功率

这样用户不会只感受到“榜单里怎么还是这些股票”，而能更清楚地看到：

- 股票池已经变大
- 但真实 quote 覆盖率还在继续提升中

## 数据流

下一阶段的数据流保持如下：

1. 本地目录文件扩容
2. 导入脚本写入 `market_symbols`
3. 市场列表和搜索从 `market_symbols` 读取
4. movers 构建任务从 `market_symbols` 批量取 symbol
5. movers 脚本抓取真实 quote
6. 成功结果写入 `market_movers_snapshots`
7. 前端页面读取：
   - `list` 分页结果
   - `movers` 快照结果

## 交付阶段

建议拆成三个连续阶段：

### Phase 1：主数据继续扩容

- 扩大 `US / HK` 目录文件
- 增强导入脚本摘要和重复执行能力
- 完成一轮实际扩容导入

### Phase 2：页面数量感知增强

- 市场页更清楚展示总数、分页、覆盖率
- 搜索结果与分页体验同步优化

### Phase 3：真实行情覆盖率补强

- 继续扩大 movers 真正覆盖的 symbol 数
- 优先补美股真实行情覆盖不足问题

## 成功标准

本次扩容完成后，应满足：

1. `market_symbols` 中的股票数量显著高于当前版本
2. 美股与港股都能在页面上体现更大的市场总数
3. `全部股票` 模式下分页浏览的股票池明显变大
4. `涨幅榜 / 跌幅榜` 的候选池和覆盖统计明显变大
5. 整个流程仍然坚持：
   - 不写模拟行情
   - 不把失败 quote 伪装成成功
   - 不牺牲页面稳定性

## 一句话结论

下一阶段不是继续打磨榜单外观，而是继续做大市场数据底盘：

**扩大 `market_symbols` 覆盖范围，工程化导入链路，并在更大的股票池上继续提升真实行情与榜单覆盖率。**
