# 全市场证券主数据与榜单分层设计

日期：2026-04-13

## 背景

当前市场模块虽然已经支持：

- `美股 / 港股` tab
- `全部 / 涨幅榜 / 跌幅榜`
- 区间筛选
- 搜索
- 榜单、曲线、订单簿三分页

但“榜单样本数量太少”的根因并不在前端，而在后端股票池来源本身还不是全市场。

目前 `/api/v1/market-data/list` 依赖两部分合并：

1. Rust 代码中的内置股票列表
2. provider 脚本临时返回的股票列表

这意味着当前市场页看到的仍然是“扩大的样例池”，而不是“可持续维护的全市场证券目录”。

如果继续沿着“往内置列表里加更多 symbol”这条路扩容，会很快遇到以下问题：

- 股票目录维护困难
- 更新不及时
- 不适合分页和筛选
- 不适合做真正的涨幅榜 / 跌幅榜
- 前端每次打开页面都可能触发过重的实时行情拉取

因此这次改造目标不是“继续加一点 symbol”，而是把市场模块升级成：

`证券主数据离线管理 + 实时行情在线拉取 + 榜单快照加速`

## 目标

本次设计目标是：

1. 建立真正可扩展的证券主数据层
2. 第一阶段覆盖：
   - 普通股票
   - ETF
3. 支持：
   - 全量可浏览
   - 榜单优先查看
   - 搜索快速定位
4. 市场页的“全部股票”支持分页浏览
5. 市场页的“涨幅榜 / 跌幅榜”支持从快照或缓存快速读取
6. 保持“目录离线更新，实时行情在线拉取”的混合模式

## 非目标

本次不做：

- 权证
- 牛熊证
- REIT
- 优先股
- ADR 单独分类治理
- 全市场实时 streaming 排名
- 多市场混合榜单
- 复杂自选股体系
- 多层缓存和复杂失效策略

重点只放在“把市场主数据体系搭起来，并让市场页真正具备全市场扩容能力”。

## 方案比较

### 方案 A：证券主数据 + 实时行情分层

推荐采用。

结构如下：

- 证券目录离线导入并落库
- 榜单和搜索基于本地主数据表
- 页面展示时只为当前页股票拉实时行情
- 涨幅榜 / 跌幅榜优先读取快照

优点：

- 架构稳定
- 扩容能力强
- 适合分页、筛选、搜索
- 不会把全市场压力直接压到前端
- 后续可继续扩到更多证券类型

缺点：

- 需要新增数据库表、导入脚本和快照任务

### 方案 B：继续扩大内置目录

优点：

- 实现快

缺点：

- 维护成本高
- 仍然不是真正的全市场方案
- 后续一定会返工到主数据表

### 方案 C：全部依赖外部 provider 动态拉取

优点：

- 本地结构最轻

缺点：

- 稳定性差
- 容易受限流影响
- 搜索、分页、榜单都难以稳定控制

## 最终设计

### 1. 数据分层

市场模块拆成三层：

1. `证券主数据层`
   - 告诉系统“市场里有哪些证券”
2. `实时行情层`
   - 告诉系统“这些证券当前价格和涨跌是多少”
3. `榜单快照层`
   - 告诉系统“当前市场的涨幅榜 / 跌幅榜结果”

这三层职责明确：

- 主数据负责覆盖范围和查询能力
- 实时行情负责价格数据
- 榜单快照负责性能和榜单体验

### 2. 新增证券主数据表

新增 `market_symbols` 表。

建议字段：

- `id`
- `symbol`
- `instrument_name`
- `market`
- `exchange`
- `country`
- `instrument_type`
- `is_active`
- `source`
- `aliases`
- `updated_at`
- `created_at`

字段说明：

- `market` 先支持：
  - `US`
  - `HK`
- `instrument_type` 第一阶段只保留：
  - `Common Stock`
  - `ETF`
- `source` 用于标记导入来源
- `aliases` 用于支持更好的搜索匹配

约束建议：

- `symbol` 全局唯一
- `(market, exchange, instrument_type)` 建组合索引
- `instrument_name`、`aliases` 为搜索预留索引能力

### 3. 主数据导入方式

采用离线导入模式。

第一阶段：

- 美股：
  - NYSE / NASDAQ 股票名录
  - 主流 ETF 名录
- 港股：
  - HKEX 上市普通股票
  - HKEX ETF

导入方式：

- 增加一个主数据导入脚本
- 支持全量导入
- 支持增量覆盖更新
- 支持停牌/退市后将 `is_active` 标记为 false

更新频率：

- 目录数据：
  - 每日一次或每周一次都可接受
- 不要求盘中实时更新目录

### 4. 实时行情策略

实时行情继续沿用现有 provider 模式，不改成全市场全量拉取。

原则：

- 页面只为当前页可见证券拉实时行情
- “全部股票”模式只拉当前分页范围内的行情
- 搜索结果页只拉命中的少量 symbol
- 曲线页和订单簿页继续只拉单只股票

这样可以避免：

- 首页打开就拉几千只股票行情
- provider 限流
- 页面首屏过慢

### 5. 榜单快照层

新增市场榜单快照机制。

用途：

- 给 `涨幅榜 / 跌幅榜` 提供快速结果
- 避免每次打开榜单页都对全市场重新批量取数和排序

第一阶段设计：

- 按市场存储：
  - `US`
  - `HK`
- 按证券类型存储：
  - `Common Stock`
  - `ETF`
- 按榜单方向存储：
  - `gainers`
  - `losers`

快照内容可包含：

- `symbol`
- `price`
- `change`
- `change_percent`
- `exchange`
- `currency`
- `captured_at`

更新方式：

- 先支持手动生成
- 后续再扩为定时任务

### 6. API 设计调整

#### 6.1 `GET /api/v1/market-data/list`

改为基于 `market_symbols` 表分页查询。

支持参数：

- `market`
- `instrument_type`
- `exchange`
- `search`
- `page`
- `page_size`
- `active_only`

返回内容：

- 当前页证券目录信息
- 总数
- 分页信息

这个接口只负责“证券目录”，不负责榜单排序。

#### 6.2 `GET /api/v1/market-data/movers`

新增榜单接口。

支持参数：

- `market`
- `instrument_type`
- `direction`
  - `gainers`
  - `losers`
- `limit`

返回：

- 榜单快照结果
- 快照时间
- 数据来源

#### 6.3 搜索接口

现有搜索接口改为优先基于 `market_symbols` 表。

支持：

- 代码搜索
- 名称搜索
- 别名搜索
- 市场过滤
- 证券类型过滤

### 7. 市场页行为调整

#### 7.1 全部股票

“全部”模式改为：

- 真正的全市场列表
- 分页浏览
- 支持市场过滤
- 支持证券类型过滤
- 支持搜索

不再要求一次性把整个市场全部拉到前端。

#### 7.2 涨幅榜 / 跌幅榜

“涨幅榜 / 跌幅榜”改为：

- 优先读取快照
- 默认返回前 N 名
- 保持快速打开

#### 7.3 证券类型筛选

市场页新增一级筛选：

- `普通股票`
- `ETF`

第一阶段不展示其他证券类型。

#### 7.4 搜索

搜索仍然保留在榜单页顶部，但改成：

- 基于主数据表
- 在当前市场范围内搜索
- 可按证券类型进一步过滤

### 8. 对现有三分页的影响

#### `/market`

榜单页是本次改造的核心受益页面：

- 从“有限样本榜单”升级成“可分页浏览的全市场榜单”

#### `/market/chart`

曲线页不依赖主数据分页逻辑，只需：

- 可以继续从榜单点击进入
- 可以继续通过 `?symbol=` 直达

主数据层主要给它补充：

- 更准确的股票名称
- 市场与证券类型元信息

#### `/market/orderbook`

订单簿页同样不需要大改。

它继续负责：

- 单只股票盘口展示

主数据层只为它补充：

- 更好的 symbol 元信息

### 9. 迁移策略

为了降低风险，建议按以下顺序迁移：

1. 新增 `market_symbols` 表与迁移
2. 写导入脚本，把现有内置 symbol 先导入进去
3. 修改 `list` 接口，让它优先从数据库读
4. 扩充导入源，逐步扩大到真正全市场
5. 新增 `movers` 快照表或快照存储
6. 新增榜单接口
7. 前端把“全部列表”改成分页，把“涨幅榜 / 跌幅榜”改成快照读取

这样即使中途停止，也能保持市场页持续可用。

### 10. 风险与注意点

#### 数据规模风险

如果全市场目录直接一次性进前端，会导致页面过重。

因此必须分页。

#### provider 压力风险

如果榜单仍然每次全量现算，会导致 provider 压力很大。

因此必须引入榜单快照。

#### 搜索一致性风险

如果搜索继续走 provider，而榜单走本地主数据，会导致体验不一致。

因此搜索也要统一到主数据表。

#### 目录更新风险

如果导入脚本不支持失效标记，会导致退市证券长期残留。

因此需要 `is_active` 字段。

## 成功标准

这次改造完成后，应满足：

1. 市场页“全部”模式不再只显示几十只样本股
2. 美股和港股都能分页浏览全市场普通股票与 ETF
3. 搜索结果与榜单来源一致
4. 涨幅榜 / 跌幅榜打开速度稳定
5. 榜单页不会因为全市场扩容而明显卡顿
6. 曲线页和订单簿页继续可用，不因主数据扩容而退化

## 一句话结论

这次不是“继续给榜单补更多 symbol”，而是把市场模块升级成：

`可维护的全市场证券主数据系统 + 按需拉取实时行情 + 快照驱动的榜单能力`
