# 项目后续开发规划 Spec

日期：2026-04-02
作者：Codex
状态：Draft

## 1. 背景

当前项目已经完成一轮围绕“私有部署、自用型量化研究 MVP”的核心建设，主链路已经基本打通：

- 创建策略
- 配置参数
- 运行回测
- 查看回测报告
- 管理回测历史与策略配置

最近一轮迭代已经补齐了以下关键能力：

- 策略名称与内部策略类型解耦
- 策略管理的编辑、删除、表单校验、模板起步、标的搜索与周期下拉
- 回测报告的参数快照、数据来源说明、策略名称/类型拆分展示
- 回测前确认摘要与更明确的失败原因反馈
- 研究效率增强，包括最近使用标的、常用标的、结果对比、JSON/CSV 导出
- 私有部署、备份恢复、健康检查与基础运维文档

这意味着项目已经从“可演示原型”进入“可持续迭代的私有研究 MVP”阶段。

接下来的开发重点不应该再平均铺新功能，而应该围绕以下原则推进：

- 先让系统长期稳定可用
- 再提升研究效率与实验能力
- 再增强数据可信度和回测解释力
- 最后为半自动执行预留稳定入口

## 2. 当前阶段判断

### 2.1 当前定位

项目当前最合理的产品定位是：

**私有部署、自用型量化研究工作台**

它的核心价值不是“直接替代完整交易平台”，而是让用户可以在统一界面中完成策略研究闭环，并对回测结果建立足够的信任。

### 2.2 当前优势

- 主链路已经成型，能够支持连续策略研究
- 数据语义比早期版本清晰，减少了“模拟数据被误解为真实数据”的风险
- 前后端的展示名称、策略类型、删除清理等基础模型更一致
- 已具备最小可维护性，部署、刷新、健康检查、备份都有了流程

### 2.3 当前短板

- 系统稳定性仍偏“个人可用”，还不到“长期省心运行”
- 研究流程仍偏单次操作，缺少真正的实验管理能力
- 回测可信度仍然以单次报告解释为主，缺少数据质量标识和系统性校验
- UI 与流程的自动化回归验证不足，后续仍可能反复出现按钮或状态接线问题
- 对真实执行记录与研究结果的关联还不够强，目前更多是并列展示而不是统一分析

## 3. 目标

下一阶段的目标不是把系统扩成“大而全”的交易平台，而是把它升级成：

**稳定可维护的私有研究系统 + 更高效的实验工作流 + 为未来执行辅助预留接口**

具体目标如下：

1. 降低维护成本，让系统长期运行更稳、更可恢复
2. 提升研究效率，让参数试验、结果对比、实验记录更高效
3. 提升结果可信度，让回测假设、数据来源、质量状态更透明
4. 为未来的“信号确认后执行”做好产品与数据结构准备

## 4. 规划原则

### 4.1 优先解决长期使用摩擦

与其继续加零散功能，不如优先消除那些会影响长期使用的痛点，比如：

- 回归 bug 难以及时发现
- 升级后不放心
- 结果对比和复盘效率低
- 报告解释不足

### 4.2 不过早进入全自动交易

当前阶段不建议直接将项目推向全自动实盘执行，因为这会显著扩大风险边界，包括：

- 风控责任增加
- 状态一致性要求更高
- 审计与恢复要求更重
- UI/后端的错误容忍空间大幅缩小

因此更合理的策略是：

先把研究系统做扎实，再逐步过渡到“研究 + 执行辅助”，而不是一步到位做全自动交易平台。

### 4.3 每个阶段都要有清晰验收标准

后续每个阶段都应该有：

- 明确用户价值
- 有限范围
- 可验证交付物
- 对后续阶段的支撑作用

## 5. 后续开发路线

### Phase 1：运行稳定性 Sprint

#### 目标

把项目从“自己能用”提升为“长期跑着也省心”。

#### 重点建设

- 升级关键前端依赖，处理已知安全更新提示
- 增加最小 UI smoke test，覆盖关键主链路：
  - 新建策略
  - 编辑策略
  - 运行回测
  - 查看/收起回测详情
  - 导出回测结果
- 将备份恢复流程从“文档说明”升级为“真实演练过的流程”
- 增加最小运维可见性：
  - 最近部署时间
  - 当前服务健康状态
  - 最近回测失败记录
- 清理现有 warning 与低风险技术债，减少后续噪音

#### 成功标准

- 核心页面具备自动化 smoke test
- 升级与部署后可快速判断系统是否健康
- 备份恢复流程真实可执行
- 已知 warning 数量下降，构建输出更干净

### Phase 2：研究工作流 Sprint

#### 目标

把当前偏“单次回测”的体验升级为更接近“实验管理系统”的体验。

#### 重点建设

- 增加参数组批量回测
- 为回测结果增加实验标签、备注、版本说明
- 将当前“结果对比”升级为“实验对比”：
  - 按策略
  - 按标签
  - 按时间
  - 按参数版本
- 增加排序与筛选增强：
  - 收益率
  - 最大回撤
  - Sharpe
  - 时间窗口
- 增加更完整的导出能力：
  - 单次回测完整快照导出
  - 多结果对比导出

#### 成功标准

- 用户可在同一策略下高效进行多次实验
- 回测结果不再只是“列表历史”，而是可组织、可筛选、可对比的研究资产
- 导出结果足以支撑离线复盘

### Phase 3：数据与可信度 Sprint

#### 目标

进一步提升回测结果的解释力、可追溯性与可信度。

#### 重点建设

- 为回测数据增加质量标识：
  - 本地数据库命中
  - 外部数据源回退
  - 数据不足
  - 缺失区间提示
- 在报告中强化“假设说明”：
  - 手续费
  - 滑点
  - 仓位约束
  - 调仓逻辑
  - 数据来源
- 增加关键场景测试：
  - 极短时间区间
  - 无行情数据
  - 数据不连续
  - 策略重命名后历史结果一致性
- 逐步强化真实执行记录与研究结果的关联语义，避免用户误读

#### 成功标准

- 用户能看懂“这次回测是基于什么数据、什么假设得出的”
- 数据异常或不足时，系统会主动提示而不是沉默失败
- 关键边界条件有测试保护，不容易在后续重构中回归

### Phase 4：半自动执行辅助 Sprint

#### 目标

在不进入全自动实盘的前提下，为未来执行辅助打基础。

#### 重点建设

- 增加策略信号确认台
- 增加“策略最新状态”概览：
  - 最近一次回测
  - 最近一次信号
  - 最近一次真实成交
- 增加回测结果与真实执行结果的并排复盘视图
- 为后续执行链路预留更清晰的数据关联方式

#### 成功标准

- 用户可以把研究结果和执行结果放在同一视图下理解
- 系统开始具备“研究指导执行”的辅助能力
- 但仍保留人工确认，不把风险直接推进到全自动交易

## 6. 推荐执行顺序

推荐优先级如下：

1. Phase 1：运行稳定性
2. Phase 2：研究工作流
3. Phase 3：数据与可信度
4. Phase 4：半自动执行辅助

原因：

- 没有稳定性，后面的效率提升会不断被回归问题抵消
- 没有研究工作流，用户很难真正高频使用系统
- 没有可信度增强，结果的使用价值会受限
- 半自动执行应该建立在前面三者已经稳固的前提上

## 7. 非目标

以下内容不属于当前阶段优先事项：

- 多用户系统
- 权限与组织协作
- 云端 SaaS 化
- 全自动实盘交易
- 复杂组合级资产配置系统
- 高频/超低延迟交易能力

这些方向并非永远不做，而是当前阶段不应抢占主路线资源。

## 8. 风险与注意事项

### 8.1 功能膨胀风险

如果同时推进太多“看起来有价值”的能力，项目容易重新回到功能分散、维护成本上升的问题。

### 8.2 语义混淆风险

回测、参考成交、真实执行、信号、策略版本这些概念一旦展示不清晰，就会再次伤害用户信任。

### 8.3 自动化覆盖不足风险

核心链路缺少 UI 级验证时，未来很容易再次出现交互按钮接线失效、状态回收不完整等问题。

### 8.4 过早实盘化风险

如果在没有充分稳定性、可信度与恢复能力的情况下推进自动执行，会显著放大系统风险。

## 9. 结论

项目下一阶段最合理的方向不是横向扩功能，而是纵向增强当前主链路，把它建设成：

**一个稳定、可信、可高频使用的私有量化研究系统**

后续开发建议按以下主线推进：

- 先稳住运行与运维
- 再提升研究实验效率
- 再增强数据解释与可信度
- 最后谨慎进入半自动执行辅助

这个顺序能最大化当前系统的投入产出比，也最符合项目现阶段“私有部署、自用研究”的真实定位。
