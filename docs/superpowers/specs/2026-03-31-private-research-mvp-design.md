# 私有部署量化研究 MVP 设计

## 背景

当前项目已经具备策略管理、回测执行、回测报告、订单与持仓、Docker 私有部署等基础能力，但整体形态仍偏“功能集合”，尚未收敛成一条稳定、可信、可持续使用的主链路。现阶段的最佳目标不是扩张到完整实盘交易平台，而是形成一个供个人私有部署、自主使用的量化研究工作台。

用户已经明确了三个关键前提：
- 产品目标偏向 `可上线 MVP`
- 核心主链路为 `创建策略 → 运行回测 → 查看报告`
- 部署形态为 `私有部署给自己用`

因此，这版设计的核心任务不是继续铺广功能，而是围绕研究闭环收敛范围、增强可信度、补齐高频交互、提升部署与日常使用稳定性。

## 产品定位

本项目下一阶段定位为：

`私有部署、自用型量化研究 MVP`

它应当满足以下目标：
- 你可以在浏览器里完成策略创建、修改、回测和结果查看，而不是依赖命令行或零散脚本
- 回测报告能够明确表达“哪些是模拟结果，哪些是真实执行记录”，避免误读
- 高频操作如编辑、删除、筛选、查看详情、重复试验等能够稳定工作
- 部署和升级足够简单，适合单人长期维护

本阶段不追求：
- 多用户协作
- 权限与账号体系
- 完整实盘自动化
- 复杂投资组合优化
- SaaS 化能力

## 核心主链路

MVP 主链路定义如下：

1. 创建策略
2. 选择交易标的与周期
3. 配置参数与基础约束
4. 发起回测
5. 查看回测报告
6. 基于结果迭代策略并保存新版本

该主链路成功的判断标准是：
- 用户能在 3 分钟内完成一次从建策略到看报告的闭环
- 回测执行结果不会出现“看起来成功，实际不可信”的语义错误
- 关键界面不依赖隐式知识，例如必须手动输入内部类型名或猜测按钮行为
- 失败状态可见，校验规则明确，历史记录可回查

## 设计原则

### 1. 研究优先，而非交易优先

当前阶段，所有产品与工程决策都应服务于研究效率和研究可信度，而不是提前为实盘交易铺过多复杂设施。凡是不能明显提升“建策略、跑回测、看报告”主链路价值的工作，都应延后。

### 2. 语义明确

系统必须清楚区分：
- 策略名称 vs 策略类型
- 模拟成交 vs 真实成交
- 回测结果 vs 实盘结果
- 停用策略 vs 删除策略

所有界面、接口与数据模型都应避免这些概念混淆。

### 3. 单机私有部署友好

设计应优先支持 Docker 本地部署或单台服务器部署。监控、日志、配置和升级都应尽量轻量，不引入只有多人协作或大规模服务才有意义的复杂架构。

### 4. 可信优于花哨

相比继续增加新页面或新策略类型，优先补齐以下能力：
- 参数校验
- 数据来源标识
- 错误提示
- 历史记录一致性
- 关键链路测试

## 功能范围

### 本阶段必做

#### 策略管理
- 支持创建、编辑、删除、启停策略
- 支持策略名称独立维护
- 交易标的支持搜索选择与手动输入补充
- 周期使用固定可选项
- 参数表单具备基础校验与默认值

#### 回测执行
- 从策略页直接发起回测
- 回测运行后写入历史
- 保存参数快照、时间范围、标的与策略信息
- 回测失败时明确提示原因

#### 回测报告
- 列表可按策略名称、策略、标的筛选
- 详情可正常展开/收起
- 明确展示策略名称、策略类型、策略 ID
- 明确区分模拟成交明细与真实成交明细
- 报告展示核心指标：收益率、年化、回撤、Sharpe、盈亏统计

#### 私有部署
- 使用现有 Docker 部署体系完成单机部署
- 保持 `refresh backend/frontend` 的增量更新能力
- 健康检查、基础日志、部署文档可用

### 本阶段不做

- 用户系统、权限系统
- 自动实盘下单闭环
- 多组合、组合对账体系
- 策略编排器和任务调度平台
- 大规模可观测性体系

## 模块设计

### 策略模块

策略模块负责：
- 维护策略元数据
- 维护可编辑参数
- 承担“策略类型”和“展示名称”的映射

下一阶段应进一步稳定其职责边界：
- `name` 只用于内部策略类型识别
- `display_name` 只用于界面展示
- `parameters` 保存研究参数
- 删除行为应视为“永久删除研究对象及其关联记录”

### 回测模块

回测模块是 MVP 的核心价值承载区。它需要做到三件事：
- 让回测执行稳定可复现
- 让报告结果清晰可读
- 让用户快速对比和迭代实验

下一阶段不扩大策略引擎范围，而是优先增强：
- 参数快照可见性
- 结果列表筛选能力
- 报告语义清晰度
- 失败原因与异常状态提示

### 真实成交数据模块

真实成交不是本阶段主链路核心，但它对“报告可信度”有帮助。因此它应作为辅助模块存在：
- 回测报告中可展示与策略匹配的真实成交记录
- 不将真实成交和模拟回测混为一谈
- 不要求本阶段形成自动实盘闭环

### 部署与运维模块

对于私有部署 MVP，部署模块目标是“简单、稳定、能恢复”：
- 一条命令启动
- 一条命令刷新前端或后端
- 有健康检查
- 有可阅读的部署 SOP

后续重点不是上云平台适配，而是：
- 配置收敛
- 数据备份恢复
- 升级流程清晰

## 数据与可信度

项目已经开始区分模拟数据和真实执行数据，这是正确方向。后续还需要继续加强：

- 在回测报告中突出显示数据来源与数据性质
- 在真实成交为空时明确说明“暂无真实记录”，而不是静默退回模拟数据
- 在策略删除时同步清理关联研究数据，避免历史悬挂记录
- 对时间范围、标的、参数快照做更完整展示，支持回测结果复盘

## 工程方向

下一阶段工程工作应围绕“稳定性、清晰性、可维护性”展开，而不是继续堆叠新能力。

优先方向：
- 巩固策略与回测主链路
- 为策略/回测关键路径增加测试
- 补齐前端交互一致性
- 统一错误提示与状态反馈
- 保持部署流程简洁

可以延后方向：
- 更复杂的券商集成
- 多页面重构
- 大规模架构拆分
- 高复杂度实时系统优化

## 分阶段路线图

### Phase 1：主链路稳定化

目标：
把“创建策略 → 运行回测 → 查看报告”做成稳定闭环。

交付重点：
- 策略表单体验收敛
- 删除、编辑、筛选、详情等交互问题清零
- 回测报告语义彻底理顺
- 回测与策略核心路径补测试

### Phase 2：研究效率增强

目标：
提升频繁试验和结果对比效率。

交付重点：
- 策略模板
- 常用标的/最近使用
- 回测结果对比
- 参数版本管理
- 报告导出

### Phase 3：可信度增强

目标：
让报告更适合长期研究使用。

交付重点：
- 更完整的回测假设配置
- 数据质量与来源状态标识
- 指标扩展
- 异常数据提示

### Phase 4：私有部署完善

目标：
让项目适合长期私有运行与升级。

交付重点：
- 备份恢复方案
- 部署/升级手册完善
- 最小监控与日志排障路径

## 风险与边界

### 风险 1：范围再次膨胀

如果在 MVP 阶段同时追求研究平台、半自动交易和完整实盘，将导致路线分散，主链路迟迟无法打磨完成。

控制方式：
- 一切优先级围绕研究主链路排序
- 任何新需求都先回答：是否直接提升“建策略、跑回测、看报告”

### 风险 2：回测语义继续混乱

如果继续把模拟结果和真实记录混在同一概念里，用户会对报告可信度失去判断。

控制方式：
- 所有相关 UI 和接口字段统一命名
- 报告显式标注数据性质

### 风险 3：工程复杂度超过私有部署收益

如果引入过多多人协作、平台化或企业级能力，会显著提高维护成本，但对当前使用场景帮助有限。

控制方式：
- 所有基础设施投入都以“单人长期私有使用”作为判断标准

## 验收标准

当以下条件成立时，可认为 MVP 达到下一阶段可用状态：

- 策略创建、编辑、删除、回测、查看报告可稳定完成
- 回测报告中不再存在明显的数据语义误导
- 私有部署后服务能稳定启动、刷新与健康检查
- 关键路径已有基础自动化校验
- 文档足以支持未来继续迭代

## 结论

这个项目当前最合理的发展方向，不是立刻扩展成完整交易平台，而是先成为一个你愿意反复打开、反复使用的私有量化研究工作台。

只要主链路稳定、报告可信、部署省心，这个 MVP 就已经具备真实价值。后面的所有扩展，无论是信号确认、人工下单，还是更复杂的研究能力，都应该建立在这条稳定主链路之上。
