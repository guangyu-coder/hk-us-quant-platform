# 市场页全市场目录与涨跌幅区间筛选设计

日期：2026-04-09

## 背景

当前市场页已经支持：

- `美股 / 港股` tab 切换
- 当前市场下的 `全部 / 涨幅榜 / 跌幅榜`
- 搜索结果按当前市场过滤

但仍有两个明显缺口：

1. 当前股票目录过小，更像“热门股票池”，不满足“尽可能完整的全市场股票”目标。
2. 涨跌筛选只有榜单模式，没有通用的区间筛选能力，无法直接筛 `>= 5%`、`<= -5%` 或 `-3% ~ 5%`。

本次迭代聚焦把市场页升级成更接近“全市场浏览器”的形态，同时保持现有页面结构稳定。

## 目标

本次设计只解决两个核心问题：

1. 把美股和港股的市场目录扩展到“尽可能大的内置目录”
2. 把涨跌幅筛选升级成通用区间筛选

用户完成后的使用方式应为：

- 进入市场页
- 切换 `美股 / 港股`
- 在当前市场中查看 `全部 / 涨幅榜 / 跌幅榜`
- 输入最小、最大涨跌幅区间
- 得到符合条件的股票列表

## 非目标

本次不做：

- 动态外部全量股票目录同步
- 分页
- 无限滚动
- 服务端分段懒加载
- 区间筛选结果导出
- 多条件复合高级筛选器

这些能力后续可以在目录规模和性能稳定后再补。

## 方案选择

### 方案 A：超大内置目录 + 前端区间筛选

推荐采用。

做法：

- 后端扩充美股和港股的本地目录
- 继续使用 `GET /api/v1/market-data/list?market=US|HK`
- 前端基于批量 quote 结果做排序与区间筛选

优点：

- 风险最低
- 对现有 API 和页面结构冲击最小
- 可以最快把“市场不够全”和“不能按区间筛”两个问题一起解决

缺点：

- 全市场能力仍受本地目录规模限制
- 首屏拉取更大股票池后，加载时间会增加

### 方案 B：超大内置目录 + 服务端区间筛选

优点：

- 后续更容易扩展分页、导出和服务端排序

缺点：

- API 变更更多
- 本轮复杂度高于实际需要

### 方案 C：动态外部目录 + 分段加载

优点：

- 最接近真正意义上的全市场

缺点：

- 数据源依赖更强
- 首轮实现复杂度明显过高

## 最终设计

### 1. 数据目录扩容

后端将扩充市场目录静态数据：

- 美股：从现有少量样本扩展到一份大得多的常见股票目录
- 港股：从现有少量样本扩展到更完整的主流港股目录

目录数据仍按当前结构返回：

- `symbol`
- `instrument_name`
- `exchange`
- `country`
- `instrument_type`
- `aliases`

目录的设计原则：

- 优先覆盖用户最常查询的大盘、科技、金融、消费、ETF 等标的
- 保证港股代码规范化为 `.HK`
- 保持市场归属判断稳定，不把美股/港股混入对方市场

### 2. 页面筛选模型

市场页将保留当前筛选结构，并新增区间筛选：

- 市场维度：`美股 / 港股`
- 榜单维度：`全部 / 涨幅榜 / 跌幅榜`
- 区间维度：
  - `最小涨跌幅`
  - `最大涨跌幅`

区间规则：

- 两个输入都可留空
- 输入值以百分比数字表示，不带 `%`
- `最小` 只限制下边界
- `最大` 只限制上边界
- 两者同时存在时，筛选闭区间

示例：

- `最小 = 5`，等价于 `changePercent >= 5`
- `最大 = -5`，等价于 `changePercent <= -5`
- `最小 = -3, 最大 = 5`，等价于 `-3 <= changePercent <= 5`

若用户输入 `最小 > 最大`：

- 前端直接提示区间无效
- 列表不应用该次非法区间

### 3. 排序与筛选顺序

为保证心智清晰，市场页按以下顺序处理列表：

1. 载入当前市场的股票目录与 quote
2. 按榜单模式排序
3. 应用涨跌幅区间筛选
4. 渲染最终列表

说明：

- `全部`：保留目录原顺序，然后应用区间筛选
- `涨幅榜`：先按 `changePercent` 降序，再应用区间筛选
- `跌幅榜`：先按 `changePercent` 升序，再应用区间筛选

这样用户在 `涨幅榜` 模式下输入阈值时，仍会看到按涨幅排序的结果。

### 4. 界面调整

右侧市场列表区新增一个轻量筛选条：

- `最小涨跌幅`
- `最大涨跌幅`
- `重置筛选`

交互原则：

- 默认空值，表示不过滤
- 输入变更后即时生效
- 无结果时显示：
  - `当前筛选下暂无股票`
- 非法区间时显示：
  - `最小涨跌幅不能大于最大涨跌幅`

榜单按钮与区间筛选并存，不互相替代。

### 5. 搜索行为

搜索仍然只针对当前市场。

区间筛选不作用于顶部搜索下拉结果，只作用于右侧市场股票列表。

原因：

- 搜索是“查找具体标的”
- 区间筛选是“浏览当前市场表现”

两者分工不同，混在一起会让结果难以理解。

### 6. 性能边界

由于本轮选择“尽可能多”的内置目录，性能策略只做最小控制：

- 继续使用现有批量行情接口
- 保留页面刷新按钮
- 保留定时刷新
- 首屏允许比当前略慢

本轮不引入分页和懒加载，但会保持以下边界，方便后续扩展：

- 目录获取逻辑与列表变换逻辑分离
- 区间筛选收敛到 helper 中，便于后续移到服务端

## 测试设计

### 前端 helper 测试

新增或扩展 helper 测试覆盖：

- 区间筛选只填最小值
- 区间筛选只填最大值
- 区间筛选同时填最小和最大
- 非法区间拦截
- 榜单排序和区间筛选组合

### 前端 smoke 测试

扩展市场页 smoke：

- 切换 `美股 / 港股`
- 切换 `全部 / 涨幅榜 / 跌幅榜`
- 输入涨跌幅区间后列表正确变化
- 重置筛选后恢复

### 后端测试

扩展市场目录测试：

- `market=US` 返回美股目录
- `market=HK` 返回港股目录
- 更大的目录不会破坏市场归类逻辑

## 验收标准

满足以下条件即视为完成：

1. 市场页可在 `美股 / 港股` 间切换
2. 每个市场下都可查看 `全部 / 涨幅榜 / 跌幅榜`
3. 用户可以输入最小和最大涨跌幅做区间筛选
4. 非法区间会给出提示
5. 股票目录明显大于当前版本
6. 搜索结果仍按当前市场过滤
7. 前端 type-check、lint、smoke 通过

## 风险与后续

本轮主要风险有两个：

1. 更大的内置目录会让首屏批量行情请求变慢
2. 股票池越大，未来越需要分页或分段加载

这两个问题本轮接受，作为后续迭代前提。

后续优先方向：

1. 把区间筛选下沉到后端
2. 为大目录增加分页或分段加载
3. 引入动态市场目录同步，而不是只依赖静态内置列表
