# 组合回测 V1 设计

日期：2026-04-17

## 背景

当前系统已经完成了以下几条主线能力建设：

- 策略管理
- 单策略回测
- 回测报告与实验工作流
- 信号确认与人工处理队列
- 市场主数据与榜单体系

这些能力已经让项目从“私有研究 MVP”进入“可持续迭代的研究系统”阶段。

但当前研究能力仍然有一个明显边界：

**回测对象仍然基本停留在“单策略 / 单标的”层面。**

这意味着即使用户已经可以：

- 管理多个策略
- 管理多个标的
- 对单次回测做对比
- 导出回测结果

也还不能直接回答更接近真实资产配置的问题，例如：

- 如果把资金按固定比例分配到 5 只股票，结果会怎样？
- 如果每周把权重调回目标比例，回撤和收益会如何变化？
- 组合层面的收益曲线与单标的回测有什么差异？
- 调仓频率对组合表现的影响是什么？

因此，下一阶段最有价值的研究能力增强不是继续堆单策略功能，而是补上：

**组合回测 V1：固定权重、多标的、周期再平衡**

## 目标

本次设计目标是：

1. 支持创建一个由多只标的组成的固定权重组合
2. 支持按固定频率再平衡：
   - `daily`
   - `weekly`
   - `monthly`
3. 支持对组合运行回测并生成独立报告
4. 输出组合层面的关键指标：
   - 累计收益
   - 年化收益
   - 最大回撤
   - Sharpe
   - 权益曲线
5. 输出组合层面的调仓记录与持仓快照
6. 不破坏现有单策略回测语义与数据结构

## 非目标

本次不做：

- 多策略信号汇总后形成组合
- 风险平价
- 最优化权重
- 阈值再平衡
- 组合嵌套策略
- 行业/风格暴露分析
- 基准指数对比
- 多币种组合
- 税费模型细化
- 现金管理和利息模拟
- 整数股最优分配

第一版只聚焦在：

**固定权重、周期再平衡、组合级回测报告**

## 方案比较

### 方案 A：固定权重 + 周期再平衡

推荐采用。

结构如下：

- 用户配置多个标的及目标权重
- 用户设定再平衡周期
- 系统按回测区间逐步推进
- 在每个调仓点把实际权重调整回目标权重
- 输出组合级权益曲线、调仓记录和指标

优点：

- 边界最清楚
- 最接近“组合回测第一版”的核心价值
- 易于理解和验证
- 前后端数据模型都更稳定
- 后续容易扩展到阈值再平衡或策略驱动组合

缺点：

- 不能表达更复杂的动态组合逻辑

### 方案 B：固定权重 + 阈值再平衡

优点：

- 更贴近真实资产管理
- 调仓次数更可控

缺点：

- 第一版就会引入更多规则和更多边界条件
- UI 配置和回测解释复杂度明显上升

### 方案 C：多策略驱动组合

优点：

- 想象空间更大
- 与现有策略系统衔接更强

缺点：

- 范围会迅速膨胀
- 需要先解决信号优先级、冲突、组合级风险约束等问题
- 不适合第一版落地

## 最终设计

### 1. 产品定位

组合回测 V1 的定位是：

**面向研究场景的多标的固定权重组合回测工具**

用户可以：

- 选择多个标的
- 设置各自目标权重
- 配置回测区间和交易成本
- 选择再平衡频率
- 查看组合层面的收益与调仓结果

它不是策略编排器，也不是实盘组合管理器，而是一个面向研究的组合级回测能力补足。

### 2. 产品入口

建议在现有回测能力旁边新增独立入口，而不是把组合配置硬塞进单策略页。

推荐结构：

- 保留现有单策略回测入口
- 新增 `组合回测` 入口页
- 组合回测拥有自己的配置页和报告页

这样可以避免以下问题：

- 单策略表单被迫混入多标的配置
- 现有回测数据模型被污染
- 前端逻辑里出现大量“如果是组合就……”分支

### 3. 组合配置模型

组合回测配置至少包含：

- `name`
  - 组合名称
- `description`
  - 可选说明
- `universe`
  - 标的列表
- `weights`
  - 每个标的的目标权重
- `initial_capital`
  - 初始资金
- `fee_bps`
  - 手续费
- `slippage_bps`
  - 滑点
- `rebalancing_frequency`
  - `daily / weekly / monthly`
- `start_date`
  - 回测起始日期
- `end_date`
  - 回测结束日期
- `is_active`
  - 是否启用

其中 `universe` 中的每一项建议至少包含：

- `symbol`
- `display_name`
- `target_weight`
- `market`
- `instrument_type`

### 4. 权重校验规则

第一版必须强制约束权重合法性。

建议规则：

- 所有权重必须大于 0
- 权重总和必须等于 1
- 允许前端以百分比输入，但后端统一存小数
- 保存前必须做精度容忍校验，例如允许极小浮点误差

错误提示必须明确，例如：

- `目标权重总和必须等于 100%`
- `每个标的的目标权重必须大于 0`
- `组合至少需要 2 个标的`

### 5. 回测执行模型

组合回测 V1 的核心逻辑如下：

1. 读取组合内全部标的的历史行情
2. 对齐回测区间内的价格时间轴
3. 初始日按目标权重分配资金
4. 在每个再平衡点：
   - 读取当前各标的市值
   - 计算当前权重
   - 计算目标调仓金额
   - 应用手续费和滑点
   - 调整持仓
5. 在非调仓日，仅根据价格变化更新组合净值
6. 最终输出组合级指标和明细

### 6. 时间轴与价格对齐

多标的组合回测的第一版必须定义清楚时间轴规则。

建议规则：

- 组合使用统一的回测频率，先限定为 `1d`
- 只支持日线级组合回测
- 对于个别标的缺失某日价格：
  - 优先使用最近可用收盘价前值填充
  - 若在起始阶段没有足够价格，则认为该标的不满足回测条件并直接报错

这样可以显著降低第一版复杂度。

也就是说，第一版不支持：

- 组合内混合不同 timeframe
- 分钟级组合回测
- 盘中再平衡

### 7. 调仓规则

第一版采用：

**固定周期、全量调回目标权重**

例如：

- `daily`
  - 每个交易日收盘后调仓
- `weekly`
  - 每周一个固定交易日调仓
- `monthly`
  - 每月一个固定交易日调仓

为了降低解释成本，建议统一采用：

- 用当日可用价格进行调仓估值
- 扣除手续费和滑点后更新持仓

### 8. 报告输出

组合回测报告应至少包含 4 个部分。

#### 8.1 组合摘要

- 组合名称
- 回测区间
- 标的数
- 再平衡频率
- 初始资金
- 手续费 / 滑点

#### 8.2 组合绩效指标

- 累计收益
- 年化收益
- 最大回撤
- Sharpe
- 波动率
- 调仓次数

#### 8.3 权益曲线与持仓概览

- 组合权益曲线
- 各标的目标权重表
- 期末持仓权重

#### 8.4 调仓记录

每条调仓记录建议包含：

- 调仓时间
- 标的
- 调仓前权重
- 目标权重
- 调仓后权重
- 买入/卖出方向
- 调仓数量或金额
- 手续费
- 滑点成本

### 9. 数据模型

建议新增独立的数据结构，而不是复用现有 `backtest_runs`。

推荐新增以下表：

- `portfolio_backtest_configs`
  - 组合配置
- `portfolio_backtest_runs`
  - 单次组合回测结果
- `portfolio_backtest_holdings`
  - 组合持仓快照
- `portfolio_backtest_rebalances`
  - 调仓记录

设计原则：

- 单策略回测与组合回测并列存在
- 不共享“同一条 run 语义”
- 前端 API 与类型也分开建模

这样可以保持：

- 现有单策略回测不被破坏
- 组合逻辑更清晰
- 后续扩展到策略驱动组合时更自然

### 10. 页面结构

建议新增一组组合回测页面。

推荐页面职责如下：

- `/portfolio-backtest`
  - 组合回测列表与入口
- `/portfolio-backtest/new`
  - 新建组合回测配置
- `/portfolio-backtest/:id`
  - 查看组合配置详情
- `/portfolio-backtest/:id/run`
  - 发起组合回测
- `/portfolio-backtest/runs/:runId`
  - 查看组合回测报告

如果当前项目更偏向扁平结构，也可以先合并为：

- 一个列表页
- 一个配置弹窗或配置页
- 一个报告页

但无论如何，必须保证：

- 与单策略回测页面职责分开
- 组合配置页面能完整表达多标的和权重
- 报告页面能完整表达组合层面的指标和调仓记录

### 11. 前端交互原则

组合配置页需要比当前单策略表单更强调“结构化输入”。

建议：

- 通过“添加标的”逐项构建组合
- 每项展示：
  - 标的代码
  - 标的名称
  - 市场
  - 目标权重
- 页面底部实时显示：
  - 当前总权重
  - 是否等于 100%
- 若不合法，禁止提交

权重输入建议：

- 默认按百分比输入
- 前端内部可临时用整数/小数百分比处理
- 提交时统一转换

### 12. 错误处理

组合回测第一版必须把错误语义做清楚。

至少需要覆盖：

- 标的数量不足
- 权重不合法
- 时间区间非法
- 历史数据不足
- 部分标的无数据
- 数据源失败

错误提示必须具体说明是哪只标的或哪类数据出了问题，避免出现泛泛的：

- `回测失败`
- `未知错误`

### 13. 测试策略

组合回测 V1 至少需要以下测试保护。

#### 13.1 后端单元测试

- 权重总和校验
- 再平衡频率解析
- 调仓金额计算
- 手续费 / 滑点计入
- 缺失数据处理

#### 13.2 后端集成测试

- 两标的固定权重组合回测
- 周度再平衡场景
- 月度再平衡场景
- 单个标的无历史数据时报错

#### 13.3 前端测试

- 标的增删
- 权重输入与总和校验
- 非法状态禁止提交
- 组合报告渲染
- 调仓记录展示

### 14. 与现有系统的关系

这次组合回测设计应被视为：

**研究系统深度升级的第一阶段**

它和现有模块的关系是：

- 不替代单策略回测
- 不替代信号确认台
- 不进入真实执行
- 只增强研究能力

未来如果继续扩展，最自然的下一步可能是：

- 阈值再平衡
- 基准对比
- 策略驱动组合
- 组合级信号和风险约束

但这些都应建立在本次固定权重 V1 跑稳之后。

## 成功标准

本次设计落地后，用户应能做到：

1. 创建一个包含多只股票的固定权重组合
2. 配置回测区间、初始资金、手续费、滑点与再平衡周期
3. 成功运行组合回测
4. 在页面中查看组合级收益、回撤、Sharpe、权益曲线
5. 查看清晰的调仓记录与组合持仓变化

如果这些都能稳定达成，就说明系统已经从“单策略研究工具”迈向了“真正可用的组合研究工作台”。
