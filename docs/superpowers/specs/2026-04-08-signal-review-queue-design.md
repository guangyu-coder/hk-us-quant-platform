# 信号留痕与待处理队列 MVP Spec

日期：2026-04-08
作者：Codex
状态：Draft

## 1. 背景

当前系统已经具备：

- 按策略生成最新信号快照
- 在交易页展示“待确认信号”
- 按信号预填订单，并保留人工确认边界
- 在回测页复盘区展示最新信号快照

但这条链路仍然缺少一个关键能力：

**信号是可见的，但不可处理。**

用户现在不能：

- 知道某条信号是否已经看过
- 标记某条信号“已确认”或“忽略”
- 给信号加备注
- 区分“最新快照”和“待处理事项”

这会导致交易页更像一个实时看板，而不是一个可持续使用的人工确认台。

## 2. 目标

本次要把“最新信号快照”升级成：

**可留痕、可处理、可备注的待处理信号队列。**

用户可以在交易页中：

1. 看到待处理信号列表
2. 对信号执行“确认 / 忽略”
3. 为信号填写备注
4. 保持“人工确认，不自动下单”的边界

## 3. 不做什么

本次不做：

- 自动下单
- 多级审批流
- 多用户协作
- 完整事件溯源系统
- 信号版本树
- 与券商订单强绑定的强审计链路

## 4. 用户体验

### 4.1 交易页新增“待处理信号队列”

交易页中的“待确认信号”模块升级为真正的待处理列表。

每条记录至少展示：

- 策略名称
- 标的 / 周期
- 信号方向 / 强度
- 生成时间
- 当前处理状态
- 备注摘要

### 4.2 记录状态

最小状态集：

- `pending`：待人工处理
- `confirmed`：已人工确认
- `ignored`：已忽略

### 4.3 操作

每条待处理信号支持：

- `按此信号预填订单`
- `标记为已确认`
- `忽略`
- `编辑备注`

说明：

- “标记为已确认”只代表人工确认了这条信号，不代表订单已发送
- “按此信号预填订单”仍然只是把数据带入订单表单
- “忽略”表示这条信号不再作为待处理项展示

### 4.4 默认列表策略

交易页默认展示：

- 当前仍为 `pending` 的信号
- 最近少量 `confirmed / ignored` 的信号作为处理历史摘要

这样既能聚焦待办，也能保留最近处理痕迹。

## 5. 数据模型

新增一张最小表，建议命名：

- `signal_reviews`

字段建议：

- `id`
- `strategy_id`
- `strategy_name`
- `symbol`
- `timeframe`
- `signal_type`
- `strength`
- `generated_at`
- `source`
- `confirmation_state`
- `note`
- `status`
- `user_note`
- `suggested_order`
- `created_at`
- `updated_at`

设计原则：

- 每次刷新信号时生成或更新一条 review 记录
- 对同一策略的“最新同类信号”可采用 upsert / replace latest 语义
- 不做完整历史爆炸式累积

## 6. 后端设计

### 6.1 刷新信号时同步落库

当调用：

- `POST /api/v1/strategies/:strategy_id/signals/refresh`
- `GET /api/v1/signals/latest`

后端在生成快照后，应同步把结果写入 `signal_reviews`。

规则：

- 新信号默认 `status = pending`
- 如果只是重复刷新出相同信号，可更新已有 pending 记录
- `confirmed / ignored` 的旧记录保留，不被覆盖

### 6.2 新增接口

新增最小接口：

1. `GET /api/v1/signals/reviews`
   - 支持按 `status / strategy_id / limit` 过滤

2. `POST /api/v1/signals/reviews/:id/confirm`
   - 将状态改为 `confirmed`

3. `POST /api/v1/signals/reviews/:id/ignore`
   - 将状态改为 `ignored`

4. `PATCH /api/v1/signals/reviews/:id/note`
   - 更新人工备注

## 7. 前端设计

### 7.1 交易页

交易页“待确认信号”模块升级为两段：

- `待处理`
- `最近处理`

待处理卡片上直接提供：

- `按此信号预填订单`
- `标记已确认`
- `忽略`

并提供一个轻量备注输入或编辑弹层。

### 7.2 回测页

回测页不改成操作台。

只需要在复盘区补充：

- 若该策略最近有 `confirmed / ignored` 的信号处理结果，可显示最近一条处理状态摘要

## 8. 成功标准

满足以下条件即可认为这一版完成：

1. 刷新信号后，系统会保留待处理记录
2. 用户可以确认、忽略、备注
3. 交易页待处理列表会随操作更新
4. 仍然明确保留“人工确认，不自动下单”
5. 不破坏现有回测、交易、策略主链路

## 9. 范围控制

这一版必须刻意克制：

- 只做单机自用型信号处理队列
- 不把“确认”包装成“已执行”
- 不要求和订单做强事务绑定

一句话定义：

**这是人工确认台的留痕版，而不是自动交易审批系统。**
