# 信号确认台最小可用版 Spec

日期：2026-04-08
作者：Codex
状态：Draft

## 1. 背景

当前项目已经完成“研究闭环”和“半自动执行辅助基础”的前两步：

- 策略可创建、编辑、删除、回测
- 回测详情已能区分模拟结果、参考执行、真实执行
- 交易页与回测页已经接入策略维度的状态概览
- 后端已提供 `/api/v1/strategies/:strategy_id/state`，可按策略返回最近回测、最近真实成交和信号占位结构

但现在的“最近信号”仍然只是占位，不具备真正的使用价值。用户还不能：

- 主动生成策略当前最新信号
- 在交易页集中看到“哪些信号值得人工确认”
- 按某条信号快速预填订单，再由人工最终提交

因此下一步的合理目标不是直接做自动交易，而是补一个边界清晰、可控的“信号确认台最小可用版”。

## 2. 目标

本次只解决一个问题：

**让用户可以生成并查看策略当前最新信号，并将信号快速带入订单表单，由人工确认后提交。**

本次不做：

- 自动下单
- 信号审批流
- 信号备注/忽略/归档
- 完整信号历史事件流
- 多用户协作

## 3. 产品定位

这一版不是“执行系统”，而是“研究指导执行”的桥接层。

用户使用方式应当是：

1. 在交易页打开信号确认区
2. 首次加载时看到全部活跃策略的最新信号快照
3. 选择当前关注的策略后，只自动刷新该策略
4. 对某条信号点击“按此信号预填订单”
5. 在订单表单中人工检查标的、方向、数量、订单类型后再提交

系统必须始终明确：

- 信号是研究辅助，不代表自动执行
- 下单动作仍由人工发起
- 没有被持久化为完整信号流的部分，不能伪装成已有完整审计记录

## 4. 方案选择

本次采用：

**A. 轻量轮询版**

### 4.1 为什么选 A

- 范围最可控，适合当前阶段
- 能最快提供真实可用价值
- 不需要一次性引入“信号历史表 + 状态流转 + 操作日志”的完整复杂度
- 与现有交易页和状态概览接口兼容性最好

### 4.2 不选 B/C 的原因

不选“事件日志版”：

- 会显著扩大范围，需要设计新的信号历史模型和状态管理
- 当前阶段用户要的是“可确认的最新信号”，不是信号运营后台

不选“纯前端即算版”：

- 刷新即丢，状态不稳定
- 无法成为交易页的可靠确认入口
- 不利于后续逐步升级成确认台

## 5. 功能范围

### 5.1 首屏全量信号快照

交易页首次打开时：

- 拉取全部活跃策略
- 为每个活跃策略生成一条“最新信号快照”
- 展示在“待确认信号”区域

这个区域至少展示：

- 策略名称
- 标的
- 周期
- 信号方向
- 信号强度
- 生成时间
- 来源说明
- 人工确认状态说明

### 5.2 当前策略自动刷新

在首次全量加载后：

- 页面只自动刷新当前选中策略的信号
- 不持续轮询全部活跃策略
- 保留“立即刷新”按钮供人工触发

这是为了在可用性和系统负载之间保持平衡。

### 5.3 人工确认预填订单

每条待确认信号提供：

- `按此信号预填订单`

点击后：

- 打开交易页现有的订单表单
- 自动带入：
  - `symbol`
  - `side`
  - `strategy_id`
- 数量采用“建议值 + 可编辑”的方式，不强行直接下单

建议值可先使用最小规则：

- 若信号中包含建议数量则使用
- 若没有则回退到默认数量 `100`

### 5.4 空状态和提示语义

系统必须明确处理以下情况：

- 没有活跃策略
- 策略可用但暂无信号
- 信号生成失败
- 行情不足导致无法生成

这些状态要显示成用户可理解的文案，而不是空白或泛化错误。

## 6. 后端设计

### 6.1 数据模型

新增最小信号快照结构，不引入完整历史流：

- `StrategySignalSnapshot`
- `StrategySignalSnapshotCollection`

最小字段建议包含：

- `strategy_id`
- `strategy_name`
- `symbol`
- `timeframe`
- `signal_type`
- `strength`
- `generated_at`
- `source`
- `confirmation_state`
- `note`
- `suggested_order`

其中 `suggested_order` 仅为预填订单使用，至少包含：

- `symbol`
- `side`
- `quantity`
- `strategy_id`

### 6.2 API 设计

新增两个层次的接口：

1. `GET /api/v1/signals/latest`
   - 按需返回全部活跃策略的最新信号快照
   - 支持可选过滤：
     - `strategy_id`
     - `limit`

2. `POST /api/v1/strategies/:strategy_id/signals/refresh`
   - 立即为指定策略刷新最新信号
   - 返回该策略的最新信号快照

`/api/v1/strategies/:strategy_id/state` 保持不变，但它的 `recent_signal` 可以开始复用真实信号快照，而不是固定 placeholder。

### 6.3 信号生成方式

本次不引入后台持续任务，采用“按请求生成”的策略：

- 首次全量拉取时，为每个活跃策略即时生成
- 单策略刷新时，为该策略即时生成

生成逻辑应复用现有策略信号引擎，而不是前端自行推导。

### 6.4 持久化策略

本次只持久化“最新快照”，不持久化完整信号历史。

可接受的实现方式：

- 单独表存每个策略最新一条信号快照
- 或先使用轻量缓存/表结构，只保留 last-write-wins 语义

关键原则：

- 后续可扩展到完整信号历史
- 当前不要引入复杂状态机

## 7. 前端设计

### 7.1 交易页结构

在 [frontend/src/app/trading/page.tsx](/Users/liguangyu/githubProj/hk-us-quant-platform/frontend/src/app/trading/page.tsx) 中新增：

- `待确认信号` 模块
- 每个信号显示卡片或列表项
- 顶部提供：
  - 首次全量加载摘要
  - 当前策略自动刷新状态
  - 手动刷新按钮

### 7.2 交互规则

进入页面时：

- 先加载全部活跃策略的最新信号

当用户点击某个策略或某条信号后：

- 将其视为当前关注策略
- 后续自动刷新只针对这个策略

点击 `按此信号预填订单` 时：

- 打开现有订单表单
- 预填表单值
- 在表单顶部显示一段提示，明确这是“基于研究信号的人工确认下单”

### 7.3 回测页联动

在 [frontend/src/app/backtest/page.tsx](/Users/liguangyu/githubProj/hk-us-quant-platform/frontend/src/app/backtest/page.tsx) 中，当前“研究与真实执行复盘”区域继续保留，但对 `recent_signal` 的展示从 placeholder 升级为真实信号快照。

这样回测页和交易页可以共用同一套信号摘要语义。

## 8. 刷新策略

采用你确认过的 `C` 方案：

- 打开页面先拉全部活跃策略的最新信号
- 后续自动刷新只盯当前选中策略
- 始终保留手动刷新入口

推荐第一版轮询策略：

- 当前策略自动刷新间隔：15s 到 30s

原因：

- 既能体现“近实时辅助”
- 又不会给本地部署环境带来明显负担

## 9. 错误处理

### 9.1 后端

需要明确区分：

- 策略不存在
- 策略未激活
- 行情数据不足
- 策略信号生成失败

错误信息必须可直接展示给前端，不应只返回模糊 500。

### 9.2 前端

交易页要分别处理：

- 全量信号加载失败
- 当前策略刷新失败
- 预填订单失败

其中“预填订单失败”通常不应阻断页面，只需给出明确提示。

## 10. 测试策略

### 10.1 后端

至少覆盖：

- 活跃策略存在时返回最新信号快照
- 无活跃策略时返回空集合
- 单策略刷新返回可用结构
- `recent_signal` 不再错误混同为真实执行

### 10.2 前端

至少覆盖：

- 首次进入交易页加载全部信号
- 切换到当前策略后仅刷新单策略
- 点击 `按此信号预填订单` 后表单被正确带值
- 无信号和错误状态可正常展示

### 10.3 Smoke Test

新增主链路：

1. 打开交易页
2. 看到待确认信号
3. 点击 `按此信号预填订单`
4. 确认订单表单已带入标的/方向/策略

## 11. 风险与边界

### 11.1 当前仍不是自动执行

即使引入“待确认信号”，也不意味着系统进入自动交易。

必须在文案和交互上持续强调：

- 信号只是建议
- 订单由人工确认提交

### 11.2 最新快照不等于历史审计

本次只做“最新信号快照”。

因此它适合作为确认台入口，但不适合作为完整审计材料。未来如果要支持：

- 忽略
- 备注
- 审批
- 历史追踪

则需要引入完整信号历史模型。

## 12. 验收标准

本次完成后，用户应该能够：

1. 在交易页看到全部活跃策略的最新待确认信号
2. 选中某个策略后，系统只自动刷新该策略信号
3. 手动点击刷新按钮重拉信号
4. 将某条信号快速带入订单表单
5. 在回测页和交易页看到一致的信号摘要语义

如果这些都成立，就说明“信号确认台最小可用版”达标。
