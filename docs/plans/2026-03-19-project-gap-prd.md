# HK-US Quant Platform 项目完善 PRD

## 1. 文档目的

基于当前仓库代码、前端页面、后端服务、测试与 CI 状态，梳理该项目从“可演示原型”走向“可持续使用的量化交易平台”仍需完善的关键能力，并形成后续 1-3 个版本的产品需求文档。

本 PRD 关注的是“还缺什么”和“为什么现在要补”，不是功能罗列。判断依据来自以下事实：

- 后端与前端主框架已具备，核心页面和 API 已经存在。
- `cargo test`、前端 `npm run lint`、`npm run type-check` 当前可通过。
- 但多个关键模块仍保留 placeholder/mock/fallback 逻辑，说明系统更接近“可运行原型”而非“可信交易系统”。

## 2. 当前状态判断

### 2.1 已经具备的能力

- Rust 后端模块边界清晰，已覆盖市场数据、策略、订单、组合、风控、事件总线、WebSocket。
- Next.js 前端已经完成仪表盘、市场、交易、组合、策略、风险、回测、设置等主要页面。
- 市场数据、策略 CRUD、订单创建、组合查询、风控指标、回测结果查询已具备基本接口。
- 基础测试、CI、Docker 化部署、迁移脚本、配置加载均已成型。

### 2.2 当前阶段的核心判断

该项目最大的缺口不是“页面不够多”，而是以下四件事还没有真正闭环：

1. 交易链路还不够可信：下单、提交、撤单、成交、持仓更新、风控审计没有完全形成可追踪闭环。
2. 数据链路还不够可信：实时行情、历史数据、降级逻辑、数据质量与新鲜度还没有产品化表达。
3. 分析链路还不够可信：回测、风险、盈亏趋势部分仍混有 placeholder 或 mock 数据。
4. 运营链路还不够可信：缺少身份权限、审计、异常告警、可运维性和真实 E2E 验证。

### 2.3 关键证据

- `src/execution/mod.rs` 中提交/撤单仍有 placeholder broker 行为。
- `src/risk/mod.rs` 中 market hours、symbol validity、VaR 等逻辑仍以简化或 placeholder 方式实现。
- `src/data/mod.rs` 仍通过 Python 脚本抓取单标的行情，数据采集与质量控制尚未形成稳定数据平台。
- `frontend/src/app/risk/page.tsx` 使用 `mockRiskHistory` 和 `mockAlerts`。
- `frontend/src/app/portfolio/page.tsx` 使用 `mockPnLData`。
- `frontend/src/components/portfolio/PortfolioSummary.tsx` 在异常时回退 `mockPositions`。
- `tests/api_test.rs` 多数测试验证的是静态 JSON 结构，而非真实 API/数据库/事件流。

## 3. 目标用户与场景

### 3.1 目标用户

- 内部量化开发/研究人员
- 交易运营人员
- 平台管理员

### 3.2 核心使用场景

- 研究员配置策略并发起回测，判断策略是否具备上线价值
- 交易员在盘中观察市场、信号、订单、持仓与风险状态
- 管理员在上游数据源异常、订单异常、风控触发时快速定位问题
- 团队在 paper trading 阶段连续运行系统，并核对系统状态与券商状态是否一致

## 4. 产品问题定义

### 问题 1：交易核心链路“能跑”，但还不“可信”

当前订单创建与状态流转已有基础，但真实 broker 提交、撤单、成交回写、持仓与盈亏同步仍未达到可靠产品要求。系统现在更像一个交易工作台原型，而不是一个可持续运行的交易执行平台。

### 问题 2：前台展示与后台真实能力不完全一致

风险页、组合页仍使用 mock 图表或 fallback 数据，这会让用户误判系统已经具备真实风险趋势和盈亏分析能力。页面“看起来完整”不等于产品能力完整。

### 问题 3：回测与风险结果缺少可信度

当回测指标、风险趋势、VaR、告警体系仍包含静态或简化逻辑时，用户无法把系统输出作为交易决策依据。这会直接削弱平台的核心价值。

### 问题 4：系统缺少内部协作所需的安全与治理能力

当前看不到真正落地的登录、角色、权限、操作审计、关键设置保存与审批机制。对于涉及交易、风险和账户的系统，这些能力不是锦上添花，而是使用门槛。

## 5. 产品目标

## 5.1 总体目标

在保持当前“模块化单体”架构不扩张的前提下，将系统从原型升级为一个可用于内部研究、paper trading 和日常演示的可信量化交易平台。

### 5.2 分阶段目标

#### V1：可信原型

- 页面展示与真实数据能力对齐
- 交易、风控、组合三条主链路可验证
- 所有 mock/fallback 行为可见且可解释

#### V2：可持续 paper trading

- 接入真实 paper broker 流程
- 完成订单与持仓对账
- 建立盘中风险监控与异常告警

#### V3：团队协作可用

- 增加账号权限、审计日志、操作追踪
- 提供稳定的回测资产、策略版本与结果沉淀
- 具备面向多角色内部用户的可运营能力

## 6. 本期优先级结论

### P0：必须先补

- 交易执行闭环
- 风控规则可信化
- 组合/PnL 真实化
- 市场数据与历史数据质量治理
- 前端移除静默 mock 伪装
- 真实集成测试与 smoke test

### P1：第二阶段必须补

- 回测结果可信化与结果管理
- 监控告警、运行状态面板、失败定位能力
- paper trading 对账与恢复机制

### P2：在系统稳定后补

- 认证鉴权
- 多角色权限
- 策略审批/版本管理
- 通知中心与协作流

## 7. 需求范围

### 7.1 需求模块 A：交易执行闭环

#### 背景

当前订单 API 与状态机已有基础，但 broker 提交/撤单仍未完全落地，订单生命周期与持仓、盈亏、告警之间的联动还不够强。

#### 需求

- 订单状态机必须完整覆盖 `Pending`、`Submitted`、`PartiallyFilled`、`Filled`、`Cancelled`、`Rejected`
- 每次状态流转必须持久化，并可用于审计与排查
- 成交后必须自动更新持仓、均价、未实现/已实现盈亏
- 撤单、拒单、部分成交都必须体现到组合与前端页面
- Mock broker 与 Alpaca broker 应遵循同一套订单契约

#### 验收标准

- 任意订单状态变化都能在 API、数据库、UI 上一致显示
- 无法出现非法状态回流
- 至少具备一条 DB-backed 的信号 -> 风控 -> 下单 -> 状态更新 -> 组合变化集成测试

### 7.2 需求模块 B：风险体系产品化

#### 背景

当前风控更多是“示意性风控”。对于交易平台来说，风控要从函数逻辑升级成用户可感知、可追踪、可解释的产品能力。

#### 需求

- 交易前风控需覆盖：订单数量限制、单标的权重限制、杠杆限制、市场时段限制、策略资金限制
- 风控失败必须返回结构化结果，而不是泛化错误
- 风险页必须展示真实风险指标来源、计算时间和触发原因
- 风险告警必须来自真实事件流，而不是页面静态 mock

#### 验收标准

- 风控拒绝时前端可见明确规则名称、阈值、触发值、严重程度
- 风险页不再依赖 `mockRiskHistory` 和 `mockAlerts`
- 至少能查询最近 N 条真实风控告警记录

### 7.3 需求模块 C：组合与盈亏真实化

#### 背景

组合页是交易平台最重要的“结果页”之一，但当前仍有 mock PnL 趋势与 fallback 持仓逻辑，说明组合分析还没有完全产品化。

#### 需求

- 组合总值、现金、持仓市值、未实现盈亏、已实现盈亏必须完全来自真实计算
- PnL 趋势图应基于真实快照或日级聚合数据
- 异常情况下页面应展示“数据不可用/降级状态”，不能静默伪装为正常

#### 验收标准

- 组合页移除 `mockPnLData`
- 前端在请求失败时不回退伪造持仓，而是展示明确错误态
- 可查询指定日期的 PnL 快照或收益曲线

### 7.4 需求模块 D：数据平台与行情可信度

#### 背景

平台现在已有行情能力，但采集方式、降级来源、历史数据一致性和新鲜度表达仍偏工程实现，尚未上升到产品级标准。

#### 需求

- 区分实时行情、降级行情、错误行情三种状态
- 历史数据需具备标准化的 symbol、interval、timezone、currency 语义
- 上游失败时需记录来源、失败原因、回退路径、数据是否 stale
- 回测与图表必须使用一致的历史数据标准

#### 验收标准

- 所有市场数据接口返回统一 metadata
- 用户能在 UI 清楚知道当前显示的是 live、degraded 还是 error
- 回测前可识别历史数据缺口，避免“带缺口回测”

### 7.5 需求模块 E：回测能力可信化

#### 背景

回测页已经有基础结果展示，但平台要真正服务量化研究，回测必须可重复、可解释、可沉淀。

#### 需求

- 回测基于持久化历史数据，不依赖临时 API 拉取
- 每次回测需记录策略版本、参数、标的、时间区间、交易成本模型
- 结果需支持查看收益率、年化、Sharpe、最大回撤、胜率、交易次数、PnL 明细
- 对同一输入重复运行应得到一致结果

#### 验收标准

- 相同参数重复运行回测，结果一致
- 回测记录可筛选、可回看、可用于对比
- 回测结果页面不出现“看起来真实但底层不可追溯”的指标

### 7.6 需求模块 F：安全、权限与治理

#### 背景

仓库中已有 settings 页的安全配置 UI，但没有看到完整认证闭环。这意味着安全能力目前主要停留在界面层。

#### 需求

- 增加登录认证与会话机制
- 区分管理员、交易员、观察者等角色权限
- 所有写操作需要鉴权
- 关键操作保留审计日志：改策略、下单、撤单、改风险阈值、执行数据生命周期任务

#### 验收标准

- 未登录用户不能访问写接口
- 角色权限可限制页面能力与 API 能力
- 能按时间与操作者查询关键操作日志

### 7.7 需求模块 G：运维与可观测性

#### 背景

系统已有日志、Docker、Prometheus/Grafana 入口，但当前还缺乏“发生问题时快速定位”的产品化运维能力。

#### 需求

- 提供服务健康、数据源健康、WebSocket 连接数、订单拒绝率、回测失败率等核心指标
- 增加 compose smoke test 或等价启动验证
- 建立最小化运行手册：迁移失败、Redis 不可用、上游行情故障、broker 不一致

#### 验收标准

- 发布前能自动跑最小回归
- 能快速判断故障位于前端、后端、DB、Redis 还是上游
- 运维文档与系统真实行为一致

## 8. 非功能需求

- 一致性：交易状态、持仓状态、风险状态必须跨 API、DB、UI 一致
- 可解释性：所有风险拒绝、数据降级、回测结果都要有明确解释
- 可追踪性：关键动作与状态变化必须可审计
- 可恢复性：上游失败或服务重启后，系统能恢复到可解释状态
- 可测试性：关键流程必须有集成测试，而不是只靠 mock 对象测试

## 9. 成功指标

### 产品指标

- 风险页、组合页、回测页 100% 移除静默 mock 数据
- 用户可在 3 分钟内定位一次订单失败的具体原因
- 用户可在 1 分钟内判断当前行情是否为真实、降级或错误状态

### 工程指标

- 至少新增 3 条 DB-backed 或服务级 E2E 流程测试
- 关键交易链路失败后，错误返回结构化、可展示
- CI 除 `cargo test`、前端 lint/type-check 外，补充至少一类运行态 smoke test

## 10. 版本规划建议

### V1：可信原型修复版（建议 2-3 周）

- 移除前端 mock 伪装
- 打通订单状态与组合同步
- 风控结果结构化
- 统一市场数据 metadata
- 增加关键集成测试

### V2：Paper Trading 准生产版（建议 3-5 周）

- 完成 Alpaca paper trading
- 建立 broker 对账与异常恢复
- 增加真实风险告警与盘中监控
- 回测结果与历史数据完全可信化

### V3：团队协作版（建议 3-4 周）

- 登录、权限、审计
- 策略版本和发布流程
- 通知与运维中心

## 11. 本期不建议优先投入的方向

- 过早扩展到微服务
- 过早接入更多券商
- 过早上机器学习策略
- 在交易与风控闭环未完成前做移动端

原因很简单：当前主要矛盾不是功能想象力不足，而是系统可信度不足。

## 12. 结论

这个项目已经越过“空壳阶段”，但还没有越过“可信平台阶段”。接下来最需要完善的不是更多页面，而是把已有页面背后的数据、交易、风控、回测和治理能力做实。

如果只做一个总判断，就是：

**从现在开始，项目的主线应该从“继续堆功能”切换为“补齐可信闭环”。**
